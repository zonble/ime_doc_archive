# Contributing

這份文件定義本 repo 的整理原則。目標是讓文件能持續擴充，同時維持來源可追溯、內容可比對、結構可維護。

## 專案目標

-   保存舊版中文輸入法說明文件
-   以現代可讀格式整理與發布
-   盡量保留原始內容、版面語氣與時代痕跡
-   明確區分「原始內容」與「整理者補充」

## 收錄範圍

優先收錄下列材料：

-   安裝程式或系統內附的說明文件
-   CHM、HLP、HTML、TXT、RTF 等原始文件轉出的內容
-   軟體附帶的圖片、操作畫面、圖示與鍵盤配置
-   能確認版本與來源的補充說明

不建議收錄下列材料，除非能清楚標註：

-   無法確認來源的二手轉貼
-   已被大幅改寫、無法回推原貌的內容
-   與輸入法本體無直接關聯的泛用教學

## 基本原則

### 1. 忠於來源

-   盡量保留原始段落順序、術語與措辭
-   原文中的時代用語、作業系統名稱、產品名詞，不主動現代化
-   若原文有明顯錯字，可修，但避免改變語意

### 2. 整理而非重寫

-   允許調整為較穩定的 Markdown 結構
-   允許補齊標題層級、清單、表格、圖片路徑
-   不為了「好讀」而任意刪減、摘要或改寫全文

### 3. 補充必須可辨識

-   整理者新增的來源說明、缺文註記、轉檔備註，應與原文分開
-   建議以前言、註解或獨立段落標示，不混入原始敘述

## 檔案與目錄規範

-   每套文件使用獨立目錄，目錄名應反映產品與版本
-   Markdown 檔名盡量使用英文或既有轉檔名稱，避免後續連結難以維護
-   圖片放在同目錄或明確的子目錄內，避免跨目錄相對路徑過深
-   如果原始文件已分章，應盡量維持相同拆分方式

## Markdown 整理規範

-   保留原有標題語意，但可調整為合法且一致的層級
-   清單與表格可重排，以提升可讀性
-   連結一律使用相對路徑
-   圖片連結需對應實際檔名，注意大小寫
-   若來源檔含內部跳轉，可改寫為 Markdown 連結

## Front Matter

若頁面有明確標題、摘要或來源，建議加入 front matter：

```yaml
---
title: 頁面標題
summary: 簡短說明
source: 原始檔名或來源描述
---
```

不是每一頁都必須補，但同一套文件內應盡量一致。

## 圖片與資源

-   優先保留原始圖片，不任意重壓或重繪
-   可接受為了站台顯示而轉成較通用格式，但應保留內容不失真
-   若圖片檔名混用大小寫，整理時應統一引用方式

## 導覽與站台

-   新增文件後，記得更新 `mkdocs.yml`
-   `docs/index.md` 應簡要說明新增文件的來源與背景
-   若某套文件內容很多，先補一個該目錄的 `index.md` 作為入口

## 來源註記

新增資料時，至少記錄：

-   軟體名稱
-   版本
-   平台
-   文件來源位置
-   取得方式

例如：

-   安裝目錄中的 `*.chm`
-   系統資料夾內附說明
-   Internet Archive 某個具體條目

## 建議工作流程

1. 先確認原始文件來源與版本。
2. 將原始內容拆出為 Markdown 與資源檔。
3. 修正連結、圖片路徑與標題結構。
4. 補上入口頁與來源說明。
5. 更新 `mkdocs.yml`。
6. 本機檢查站台是否能正常瀏覽。

## 提交前檢查

-   新增頁面是否已加入導覽
-   圖片與相對連結是否可用
-   檔名大小寫是否一致
-   是否把整理者補充和原文混在一起
-   是否留下可追溯的來源資訊
